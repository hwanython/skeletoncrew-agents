# autoresearch (karpathy) — 자율 연구 루프 방법론 다이제스트

> 출처: karpathy/autoresearch (2026). LLM 에이전트가 밤새 자율적으로 "연구"를 수행하게 만드는 최소 골격 실험.
> 이 문서는 도구 사용법이 아니라 **자율 연구 루프의 방법론**을 추출한 아카이브다.

## 0. 한 줄 요약

작은 실험 환경 + 고정된 단일 지표 + "수정 → 측정 → 개선되면 채택, 아니면 되돌림"의 무한 루프.
사람은 코드를 직접 만지지 않고, 에이전트에게 줄 **컨텍스트 문서(program.md)** 를 프로그래밍한다.

## 1. autoresearch가 실제로 하는 일

- 에이전트에게 진짜로 동작하는 작은 LLM 학습 셋업(nanochat 단순화판)을 주고, **밤새 혼자 실험**시킨다.
- 에이전트는 코드를 수정 → 5분 학습 → 결과가 좋아졌는지 확인 → 채택/폐기 → 반복.
- 아침에 사람은 실험 로그와 (잘 되면) 더 좋은 모델을 보게 된다.
- 핵심 전환: 연구자가 파이썬 파일을 직접 고치는 게 아니라, **에이전트의 행동 규약을 담은 마크다운(program.md)을 고친다.** 즉 "연구 조직 자체를 코드로 프로그래밍"한다.

## 2. 파일 구조와 역할 분리 (방법론의 핵심)

- `prepare.py` — **고정/읽기 전용.** 데이터 준비, 토크나이저, 평가 함수(`evaluate_bpb`), 상수(시간 예산, 시퀀스 길이). 에이전트가 절대 못 건드린다 → 평가의 무결성 보장.
- `train.py` — **에이전트가 수정하는 유일한 파일.** 모델 아키텍처, 옵티마이저, 학습 루프, 하이퍼파라미터 전부 자유.
- `program.md` — **사람이 수정하는 에이전트 지침서.** 에이전트의 "스킬"이자 작업 규약.
- `results.tsv` — 실험 로그(git 추적 안 함).

> 설계 원칙: **수정 가능한 면(train.py)** 과 **불변의 평가 면(prepare.py)** 을 물리적으로 분리한다.
> 에이전트가 지표 자체를 해킹하지 못하게 막는 것이 자율 루프 신뢰성의 전제다.

## 3. 측정 기준 설계 (가장 중요한 결정)

- **단일 지표: val_bpb (validation bits per byte)** — 낮을수록 좋음. vocab 크기에 독립적이라 아키텍처를 바꿔도 공정 비교 가능.
- **고정 시간 예산: 학습은 무조건 5분(벽시계).** 시작/컴파일 제외.
  - 효과 1: 모델 크기·배치·아키텍처가 달라져도 실험들이 **직접 비교 가능**.
  - 효과 2: 주어진 플랫폼에서 "5분 안에 최선"을 자동으로 탐색하게 됨.
  - 비용: 다른 컴퓨팅 환경과는 결과 비교 불가(트레이드오프 수용).
- 교훈: 자율 루프를 돌리려면 **단일·자동·조작 불가능한 비교 지표**와 **고정된 자원 예산**이 먼저 정의돼야 한다.

## 4. 핵심 실험 루프 (program.md의 LOOP FOREVER)

셋업 후 무한 반복:

1. git 상태 확인 (현재 브랜치/커밋).
2. `train.py`에 **실험 아이디어 하나**를 직접 코드로 적용.
3. git commit.
4. 실험 실행: `uv run train.py > run.log 2>&1` (출력은 파일로 리다이렉트 — tee 금지, 컨텍스트를 로그로 오염시키지 말 것).
5. 결과만 추출: `grep "^val_bpb:\|^peak_vram_mb:" run.log`.
6. grep이 비었으면 크래시 → `tail -n 50 run.log`로 스택트레이스 확인, 몇 번 시도 후 안 되면 포기.
7. `results.tsv`에 기록 (commit / val_bpb / memory_gb / status / description).
8. val_bpb가 **개선되면** 브랜치를 그대로 "전진"(커밋 유지) = **채택**.
9. **같거나 나빠지면** `git reset`으로 시작점 복귀 = **폐기**.

> 본질: 가설 1개 → 변경 → 측정 → 개선 여부로 채택/롤백. git이 곧 "실험 상태 저장소"이자 되돌리기 메커니즘.

## 5. 셋업 단계 (Setup)

1. 런 태그 합의(예: `mar5`), 브랜치 `autoresearch/<tag>` 신규 생성.
2. 범위 내 파일 통독(README, prepare.py, train.py)으로 전체 맥락 확보.
3. 데이터/토크나이저 캐시 존재 확인, 없으면 사람에게 `uv run prepare.py` 요청.
4. `results.tsv` 헤더만 초기화.
5. **첫 실행은 항상 baseline** — 코드를 그대로 돌려 기준선부터 기록.

## 6. 의사결정·판단 규칙 (프롬프트 패턴)

- **단순성 기준(Simplicity criterion):** 다른 조건이 같다면 단순한 게 낫다.
  - 0.001 개선인데 지저분한 20줄 추가 → 채택 안 함.
  - 코드를 **지우면서** 같거나 더 나아짐 → 무조건 채택(단순화 승리).
  - 개선 ≈ 0 인데 코드가 훨씬 단순 → 채택.
- **VRAM은 소프트 제약:** 의미 있는 이득엔 약간의 증가 허용, 폭발은 불가.
- **크래시 처리:** 사소(오타·import 누락)하면 고쳐 재실행, 아이디어 자체가 깨졌으면 status=crash 기록하고 넘어감.
- **타임아웃:** 실험은 ~5분. 10분 초과 시 kill → 실패 처리(폐기·복귀).

## 7. 정지 기준 — "멈추지 않는다"

- **NEVER STOP.** 셋업 이후엔 사람에게 "계속할까요?"라고 묻지 않는다. 사람은 자고 있을 수 있고, 무한히(수동 중단 전까지) 일하길 기대한다.
- **아이디어 고갈 시:** 멈추지 말고 더 깊이 생각 — 코드가 참조하는 논문 읽기, 범위 내 파일 재독으로 새 각도 찾기, 과거의 "아쉬운 근접 실패"들을 조합, 더 과감한 아키텍처 변경 시도.
- 즉 정지 조건은 "지표 수렴"이 아니라 **외부(사람)의 명시적 중단**이며, 내부적으로는 아이디어 재생성으로 루프를 지속한다.
- 단, 되감기(rewind/git reset to far back)는 **극도로 드물게** 사용 — 전진 누적을 우선.

## 8. 결과 기록 포맷 (재현성·감사 가능성)

`results.tsv` (탭 구분, 콤마 금지 — 설명에 콤마가 깨짐):

```
commit   val_bpb    memory_gb  status   description
a1b2c3d  0.997900   44.0       keep     baseline
b2c3d4e  0.993200   44.2       keep     increase LR to 0.04
c3d4e5f  1.005000   44.0       discard  switch to GeLU activation
d4e5f6g  0.000000   0.0        crash    double model width (OOM)
```

- 모든 시도(채택/폐기/크래시)를 **남김없이** 기록 → 사후 분석(analysis.ipynb)과 메타 학습 가능.

## 9. 재사용 가능한 일반 방법론 (도메인 불문)

이 패턴은 LLM 학습이 아니라 **어떤 자율 탐색 문제에도 이식 가능**하다:

1. **불변 평가면 / 가변 작업면 분리** — 에이전트가 지표를 못 건드리게.
2. **단일·자동·조작불가 지표 + 고정 자원 예산** 을 먼저 정의(공정 비교의 전제).
3. **가설 1개 단위의 원자적 변경** → 측정 → 채택/롤백을 버전관리(git)에 묶기.
4. **항상 baseline부터** 측정 후 모든 변경을 baseline 대비로 평가.
5. **모든 시도를 구조화 로그로 남기기**(채택/폐기/실패 사유 포함).
6. **자율성 규약 명문화** — 사람 확인 대기 금지, 막히면 새 아이디어 생성으로 루프 유지, 외부 중단까지 지속.
7. **단순성 우선·복잡도 비용 명시** — 작은 이득과 복잡도 증가를 저울질하는 규칙을 프롬프트에 박아두기.
8. **컨텍스트 위생** — 출력은 파일로, 핵심만 grep으로 추출해 에이전트 컨텍스트 오염 방지.
9. **사람은 코드가 아니라 "조직 규약(program.md)"을 반복 개선** — 메타 레벨에서 연구 속도 자체를 최적화.
